MANUAL DE NEGOCIO - EL JIREH BRASIL / URUGUAY
Actualizado: 2026-05-21

1. PROPOSITO DEL NEGOCIO

El Jireh Brasil / Uruguay es un servicio de remesas y operaciones financieras hacia Cuba. El objetivo es permitir que clientes fuera de Cuba paguen en una moneda disponible para ellos, principalmente BRL o UYU, y que el beneficiario en Cuba reciba valor mediante alguno de estos servicios:

- Transferencia a tarjeta CUP.
- Entrega de efectivo en punto de recogida.
- Recarga de saldo movil.
- Recarga o acreditacion de divisa en tarjetas MLC, USD o Clasica.

El valor principal del negocio es rapidez, confianza, claridad en las instrucciones y trazabilidad de cada operacion.

2. ESTADO ACTUAL DEL NEGOCIO

El negocio funciona principalmente por WhatsApp. El flujo actual combina atencion manual, grupos de operaciones y calculos basados en una hoja de precios.

Flujo operativo actual:

1. Se actualizan las tasas y ofertas en una hoja de calculo.
2. Se publican precios actualizados en el grupo o canal comercial de WhatsApp.
3. El cliente contacta por WhatsApp y selecciona el servicio deseado.
4. Se le envian instrucciones de pago y datos requeridos segun el servicio.
5. El cliente envia comprobante de pago y datos del beneficiario.
6. Se crea una orden interna.
7. La orden se envia al grupo de operaciones.
8. Un operador realiza la operacion en Cuba.
9. El operador envia comprobante o confirmacion al grupo.
10. Se envia confirmacion final al cliente.
11. La operacion finalizada se archiva o se reporta para control de ventas, ganancias y trazabilidad.

El sistema actual que se esta construyendo busca reemplazar progresivamente la parte manual de creacion, control y seguimiento de operaciones.

3. SERVICIOS OFRECIDOS

3.1 Transferencia a tarjeta CUP

Descripcion:
El cliente paga desde el exterior y el beneficiario recibe CUP en una tarjeta cubana.

Datos requeridos:

- Monto pagado por el cliente.
- Moneda de pago, por ejemplo BRL o UYU.
- Metodo de pago.
- Numero de tarjeta CUP.
- telefono_destinatario opcional: contacto del beneficiario o familiar en Cuba.
- Operador responsable.
- Cliente asociado, opcional en la etapa actual.
- contacto_id opcional para rellenar datos desde la libreta del beneficiario.

Calculo:

monto_resultado = monto_pago * tasa_final

tasa_final se congela en el pedido para conservar el historico aunque la oferta cambie despues.

Estado en el sistema:
Implementado y probado contra la API real.

Endpoint actual:
POST /pedido/transferencia

3.2 Entrega de efectivo

- Punto de recogida.
- Operador responsable.
- Cliente asociado, opcional en la etapa actual.
- contacto_id opcional para rellenar telefono_destinatario y documento_identidad_url desde la libreta del beneficiario.

Calculo:

monto_resultado = monto_pago * tasa_final

Estado en el sistema:
Implementado y probado contra la API real.

Endpoint actual:

3.3 Recarga de saldo movil

Descripcion:
El cliente paga desde el exterior y se recarga saldo movil a un numero cubano.

Modalidades actuales:

- Por paquete de saldo existente.
- Manual, enviando monto_pago y saldo_cup.

Datos requeridos:

- telefono_destinatario: numero cubano a recargar.
- Metodo de pago.
- Operador responsable.
- paquete_saldo_id, o monto_pago + saldo_cup.
- Cliente asociado, opcional en la etapa actual.
- contacto_id opcional para rellenar telefono_destinatario desde la libreta del beneficiario.

Calculo por paquete:
El sistema usa el monto y saldo definidos en el paquete activo.

Calculo manual:

tasa_final = saldo_cup / monto_pago
monto_resultado = saldo_cup

Estado en el sistema:
Implementado y probado contra la API real.

Endpoint actual:
POST /pedido/saldo

3.4 Divisa: MLC, USD o Clasica

Descripcion:
El cliente paga desde el exterior y se acredita divisa en una tarjeta cubana.

Datos requeridos:

- Monto pagado por el cliente.
- Moneda de pago.
- Metodo de pago.
- Tipo de tarjeta: MLC, USD o Clasica.
- Numero de tarjeta.
- Monto de divisa a acreditar.
- Operador responsable.
- Cliente asociado, opcional en la etapa actual.
- contacto_id opcional para rellenar tipo_tarjeta, numero_tarjeta y telefono_destinatario desde la libreta del beneficiario.

Calculo actual:

tasa_final = monto_divisa / monto_pago
monto_resultado = monto_divisa

Estado en el sistema:
Implementado, unificado con el flujo central de pedidos y probado contra la API real.

4. CICLO DE VIDA DEL PEDIDO

Estados definidos en el sistema:

- creado
- pendiente_pago
- pago_confirmado
- en_operacion
- completado
- canceladometodos-pago
- error

Flujo recomendado:

1. creado
   El cliente inicio una solicitud, pero aun no se completo la informacion o el pago.

2. pendiente_pago
   El pedido ya fue creado y queda esperando comprobante o confirmacion de pago.

3. pago_confirmado
   Se confirmo que el pago del cliente fue recibido.

4. en_operacion
   La operacion fue enviada o asignada para ejecucion.

5. completado
   El operador ejecuto la operacion y existe confirmacion.

6. cancelado
   La operacion no se realizara.

7. error
   Hubo un problema operativo, dato incorrecto, falla bancaria, rechazo, o situacion que requiere revision.

Compatibilidad actual:
El sistema acepta algunos nombres viejos y los normaliza:

- pendiente -> pendiente_pago
- en_proceso -> en_operacion
- realizado -> completado
- finalizado -> completado
- error_operacion -> error

Estado en el sistema:
Implementado con historial de cambios en PedidoHistorial.

Endpoint para cambiar estado:
PATCH /pedido/{codigo_operacion}/estado

7. OFERTAS Y SINCRONIZACION CON GOOGLE SHEETS

El negocio usa una hoja de calculo como fuente operativa de tasas y paquetes.

Servicios sincronizados actualmente:

- transferencia
- efectivo
- saldo
- mlc
- usd
- clasica

El sync lee la pestana operativa Calcular Oferta y el rango A1:L160, para no cargar hojas historicas completas ni cientos de filas que no se usan en la operacion diaria.
El sync debe cargar credentials.json desde la carpeta Backend, sin depender del directorio desde donde se arranque Uvicorn.
La verificacion de tablas y columnas ocurre al arrancar el backend; el sync no debe repetir mantenimiento de schema en cada ejecucion para no demorar el boton Actualizar.
Antes de escribir, el sync deduplica filas por servicio, moneda y minimo de pago; los paquetes de saldo se deduplican por moneda, monto de pago y saldo CUP.
El sync aplica escritura diferencial: crea lo nuevo, actualiza lo que cambio, conserva sin tocar lo igual y desactiva solo los registros google_sheet que ya no aparecen en la seccion vigente. No debe apagar y reactivar todas las ofertas en cada ejecucion.

Reglas actuales del parser:

- Detecta secciones por nombre de servicio en la columna B; acepta variantes con errores comunes como Transferecia, Efectlvo y Saido.
- Usa la fecha mas reciente por servicio como corte vigente. Las secciones antiguas del mismo servicio se ignoran aunque tengan otra moneda, para evitar publicar tasas viejas.
- Si hay mas de una seccion del mismo servicio y moneda dentro de la fecha vigente, gana la seccion que este mas abajo en la hoja.
- Detecta la moneda de pago principalmente desde la columna D de la fila de encabezado, para evitar confundir USDT con USD.
- Para transferencia y efectivo lee monto/minimo en columna D y tasa/oferta en columna K; si K esta vacia, calcula la tasa con OFERTA CUP de columna I dividido entre el monto de columna D.
- Para mlc, usd y clasica lee monto/minimo en columna D y tasa/oferta en columna K.
- Para saldo lee monto de pago en columna D y saldo CUP en columna L.
- Cada fila valida de oferta se sincroniza como una oferta independiente; solo se deduplican filas con la misma moneda y el mismo minimo exacto.
- El endpoint devuelve meta.secciones_usadas, meta.worksheet, meta.rango_leido, meta.ofertas_db, meta.saldo_db y meta.tiempos_seg para auditar que filas fueron sincronizadas y donde se consume el tiempo.

Endpoint actual:
POST /sync/ofertas

Estado actual:
Implementado.

Riesgos actuales:

- Si la hoja cambia las columnas D, I, K o L, debe actualizarse el parser.
- Si una seccion vigente no tiene fecha y existe otra del mismo servicio con fecha valida, la seccion sin fecha puede quedar ignorada por seguridad.
- Si el negocio decide mantener monedas vigentes con fechas diferentes para un mismo servicio, debe ajustarse la regla de seleccion vigente y documentarse aqui antes de usarlo en produccion.

20. EXPERIENCIA OPERATIVA DEL FRONTEND

El sistema debe priorizar uso movil para operarios poco tecnicos. La navegacion principal queda organizada asi:

- Inicio: dashboard operativo con tasas activas reales, logo, marca y accesos rapidos por servicio.
- Pedidos: bandeja/lista/kanban para seguimiento de operaciones.
- Crear: formulario por servicio, tambien accesible desde el FAB flotante.
- Reportes: resumen operativo y exportacion.
- Perfil: datos del operador y salida de sesion.

Reglas de navegacion y marca:

- El logo del sistema y el texto de marca EL JIREH en el header deben redirigir siempre al dashboard Inicio.
- El logo/texto del menu lateral movil tambien debe redirigir a Inicio y cerrar el menu.
- En vista Pedidos no debe mostrarse un boton Crear pedido dentro de la barra superior de filtros, porque la accion principal ya existe como boton flotante.
- La barra de pedidos usa busqueda a la derecha en desktop, filtros por servicio en chips visibles y filtros por estado en chips visibles.
- La vista Pedidos carga un lote reciente de pedidos y aplica filtros de estado, servicio y busqueda en el frontend para que los chips respondan sin latencia; el boton de actualizar pedidos recarga el lote desde el backend.
- Las vistas Lista y Kanban deben ordenar los pedidos por el mismo orden visual de estados; cancelados siempre queda al final cuando se muestran todos los estados.
- En movil debe existir boton de menu de tres rayitas en el header para abrir el menu lateral.

Reglas de datos del dashboard Inicio:

- Transferencia y efectivo leen solamente ofertas activas de servicios transferencia y efectivo, mostrando todas las ofertas/minimos activos de cada moneda.
- Las tasas del dashboard Inicio se agrupan primero por moneda de pago y luego por servicio; por ejemplo, transferencia pagada en BRL debe verse separada de transferencia pagada en UYU.
- El orden visual recomendado es mostrar primero todos los servicios con pago en BRL, despues todos los servicios con pago en UYU, y luego otras monedas activas.
- Dentro de cada moneda, solo se muestran cards/tablas de servicios con oferta o paquete activo; no se deben mostrar servicios vacios en estado sin tasa.
- Recargas moviles leen solamente paquetes_saldo activos agrupados por moneda_pago; no deben mezclarse con ofertas.
- Tarjetas en divisa leen ofertas activas de servicios mlc, usd y clasica agrupadas por moneda_pago; no usan paquetes_saldo ni ofertas de transferencia/efectivo.
- El endpoint GET /tasas-operativas/ es de lectura operativa para dashboard y separa ofertas, ofertas_divisa y paquetes_saldo.
- El boton Actualizar del dashboard ejecuta POST /sync/ofertas y luego recarga GET /tasas-operativas/ cuando el operador tiene permiso empresa:control_total; para otros roles solo recarga las tasas ya guardadas.

Reglas visuales:

- Mantener coherencia visual con el dashboard: cards con sombras suaves, bordes por estado, separadores claros y botones grandes.
- Los datos operativos dentro de cards, como tarjetas, telefonos, tasas y montos, deben mantenerse en una linea con truncado si no caben.
- Cancelados debe aparecer al final del orden visual de estados.
